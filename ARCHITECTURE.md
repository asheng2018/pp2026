### PayPal/Stripe 可追踪维度与对策

| 追踪维度 | PayPal JS SDK | Stripe.js / Elements | 对策 |
|----------|---------------|---------------------|------|
| window.location | 上报 | 上报 | iframe隔离 / 服务端转发 |
| document.referrer | 上报 | 上报 | Referrer-Policy + 中间跳转 |
| JS SDK 加载域名 | 上报 | 上报 | 全部通过B站域名加载 |

## 三、AB无痕跳转核心机制

### 方案一：服务端代理模式（推荐，安全性最高）

```
用户 ──► A站前端 ──► A站API ──► 调度层 ──► B站API(代理IP) ──► PayPal API ──► B站前端(支付页) ──► 用户
                         │                    │
                         └── 创建订单 ────────┘  (纯服务端,不走用户浏览器)
                         返回支付URL(iframe嵌入)
```

流程:
1. 用户在A站浏览商品、下单
2. A站后端将订单信息发送给调度层
3. 调度层选择一个B站账号，通过B站后端调用 PayPal/Stripe API 创建支付订单
4. 返回支付页面URL，在A站通过 iframe 无痕嵌入
5. 用户看到的支付页面域名是 B站域名（或自定义域名）
6. PayPal/Stripe SDK 只看到 B站的域名和IP

### 方案二：中间跳转页模式

```
A站前端 ──► B站跳转中间页 ──► PayPal支付页 ──► B站结果页 ──► 重定向回A站
             (纯静态,秒加载)
```

### 方案三：iframe 沙箱模式

```html
<iframe
  src="https://b-site.com/pay?token=xxx"
  sandbox="allow-scripts allow-forms allow-same-origin allow-top-navigation"
  referrerpolicy="no-referrer"
  style="width:100%;height:100vh;border:none;">
</iframe>
```

### 推荐：混合方案架构

```
                        ┌──────────────────┐
  用户浏览器              │   A站 (仿牌域名)   │
  ┌────────┐            │                  │
  │ A站页面 │──下单────►│  POST /api/order │
  │        │            │       │          │
  │        │            │       ▼          │
  │        │            │  调度层分配B站账号 │
  │        │            │       │          │
  │        │            │       ▼          │
  │        │            │  代理请求B站API   │
  │        │            │  POST /pay/create│
  │        │            │       │          │
  │        │◄───────────│  返回pay_token   │
  │        │            │                  │
  │        │──iframe───►│  /pay/gateway    │
  │        │  加载支付   │  (内嵌B站支付页)  │
```

## 五、账号池与轮询调度引擎

### 账号模型

```
PaymentAccount
├── id, gateway (paypal|stripe), alias
├── credentials (AES-256-GCM加密存储)
│   ├── PayPal: clientId, secret, merchantId, webhookId
│   └── Stripe: publishableKey, secretKey, accountId, webhookSecret
├── bSite: domain, fixedIP, region
├── limits
│   ├── singleMin / singleMax (单笔限额)
│   ├── dailyMax / monthlyMax / lifetimeMax (周期限额)
│   └── dailyCount (日笔数上限)
├── runtime (Redis存储)
│   ├── status: online|cooling|offline|draining|warming
│   ├── todayAmount / todayCount / monthAmount
│   ├── consecutiveFails / successRate
│   └── weight / priority / lastUsedAt
└── tags: usd_only, high_risk, vip
```

### 调度策略

| 策略 | 算法 | 复杂度 | 适用场景 |
|------|------|--------|----------|
| 加权轮询 (WRR) | atomic counter % totalWeight | O(n) | 各通道权重不同 |
| 顺序轮切 (Sequential) | 商户级别 lastIdx | O(1) | 收一单换下一个 |
| 最小使用率 (LeastUtilized) | sort by todayAmount/dailyMax | O(n log n) | 均衡负载 |
| 随机 (Random) | rand.Intn | O(1) | 简单防追踪 |

### 过滤链 (责任链模式)

```
请求 → StatusFilter → AmountFilter → DailyLimitFilter
    → MonthlyLimitFilter → CurrencyFilter → CountryFilter
    → SuccessRateFilter → ConsecutiveFailFilter → CoolingFilter
    → 候选账号池 → 策略选择 → 返回最优账号
```

### 熔断状态机

```
             ┌──────────┐
    ONLINE ──►│ 正常服务  │
             └─────┬────┘
                   │ 连续失败 >= 阈值
             ┌─────▼────┐
    COOLING──►│ 降权冷却  │──► 冷却时间到 ──► ONLINE
             └─────┬────┘
                   │ 冷却期内仍失败
             ┌─────▼────┐
    OFFLINE ─►│ 手动恢复  │──► 人工介入 ──► ONLINE
             └──────────┘
```
